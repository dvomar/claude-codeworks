# Review GitLab MR

Performs a comprehensive 3-pass code review of a GitLab merge request directly in the terminal.

## Usage

```
/mr-review-gitlab [MR_number]
```

Examples:
- `/mr-review-gitlab 166` — review MR !166 directly
- `/mr-review-gitlab` — list open MRs first, then pick one

## Instructions

This skill performs a review in 8 steps.

### Step 1: Resolve MR number

**If MR number was provided as argument**, skip to Step 2.

**If no MR number was provided**, list open merge requests:

```bash
glab mr list --state opened
```

Display the list and ask: **"Which MR would you like to review? Provide the MR number (e.g., 166)."**

Wait for the user's selection before proceeding.

### Step 2: Load MR metadata and diff

Run these commands via Bash tool:

```bash
glab mr view <N>
```

```bash
glab mr diff <N>
```

From the metadata, note: title, author, source and target branch.

### Step 3: Identify changed files and load their source code

Extract the list of changed files from the diff. For each changed file, read its **complete current content** using Read tool. This is critical — review cannot be done from diff alone, you need to see the context around changes.

### Step 4: Load relevant conventions from the knowledge base

Conventions are provided via CLAUDE.md and MEMORY.md (auto-injected into context).
For detailed conventions, Read these knowledge files generated by `/code-analyze-codebase`:
- `.claude/knowledge/tech-stack.md` — dependencies, versions
- `.claude/knowledge/architecture.md` — project structure, file placement
- `.claude/knowledge/backend.md` — API, data access, auth, validation patterns
- `.claude/knowledge/frontend.md` — components, styling, state, i18n patterns (if applicable)
- `.claude/knowledge/conventions.md` — naming, formatting, testing conventions

If knowledge files don't exist, run `/code-analyze-codebase` first.

Also read `CLAUDE.md` from the project root for general conventions.

### Step 5: Find similar existing code in the codebase

For each changed file, use Glob and Grep to find **similar existing files** in the same folder or layer (e.g., other controllers, other services, other repositories). Comparison with existing code is key for identifying inconsistencies.

**Limit for large MRs:** If the MR changes **more than 10 files**, search for similar code only for files where CRITICAL or IMPORTANT findings were identified in the first quick pass (Pass 1). For MRs with 10 or fewer files, search for all changed files.

### Step 6: Perform 3-pass review

#### Pass 1: Conventions and style

- Naming (variables, methods, classes, files) — consistent with conventions?
- File organization — one type per file, correct placement in layers
- DI registration — in `RegisterServicesExtension`, not in `Program.cs`
- Braces — required for all control structures (`if`, `for`, `foreach`, `while`)
- Indentation, formatting, line endings
- Unnecessary `using` directives, inconsistent import ordering

#### Pass 2: Design and architecture

- Dependency injection — correct usage, lifetime (Scoped/Transient/Singleton)
- Repository/Service/Validation/Command patterns — do they follow established patterns?
- SOLID principles — especially Single Responsibility and Dependency Inversion
- Async/await — correct usage, no async void, CancellationToken propagation
- Comparison with similar existing code — do MR authors do it the same way as elsewhere in the codebase?
- Layering — API layer doesn't call DAL directly, business logic not in controllers

#### Pass 3: Quality and performance

- DRY — duplicate code, missing extraction of shared logic
- Code smells — overly long methods, too many parameters, god class
- Performance — N+1 queries, unnecessary LINQ materialization, missing `AsNoTracking()`
- Security — SQL injection, XSS, improper use of user input
- Readability — is the code understandable without additional context?
- Nullable types — correct use of `?`, null value checks
- Error handling — consistent with existing patterns

### Step 7: Output structured results

Format output as follows:

```
# Review MR !<N>: <title>
Author: <author> | Branch: <source> -> <target>
Changed files: <count>

---

## CRITICAL (must be fixed before merge)

### [C1] <filename>:<line> — <brief description>
**Problem:** <what is wrong and why>
**Proposed fix:**
```<language>
<specific fix code>
```

---

## IMPORTANT (should be fixed)

### [I1] <filename>:<line> — <brief description>
**Problem:** <what is wrong and why>
**Proposed fix:**
```<language>
<specific fix code>
```

---

## SUGGESTIONS (nice to have)

### [S1] <filename>:<line> — <brief description>
**Suggestion:** <what could be improved and why>

---

- <what is well done>
- <which patterns are correctly followed>

---

- [ ] <specific action 2>

| Critical    | X     |
| Important   | X     |

```

### Step 8: Post draft comments to GitLab MR

After step 7, ask the user: **"Which findings should I post as draft comments to the GitLab MR? You can say 'all', list specific IDs (e.g., 'C1, I2, S1'), or 'none'."**

If the user selects specific findings, post only those. If 'all', post everything. If 'none', skip posting.

When the user confirms:

1. **Get `diff_refs`** from the MR for comment positioning:
   ```bash
   glab api projects/:fullpath/merge_requests/<N> | python3 -c "
   import sys, json
   refs = json.load(sys.stdin)['diff_refs']
   print(refs['base_sha'])
   print(refs['head_sha'])
   print(refs['start_sha'])
   "
   ```

2. **For every CRITICAL, IMPORTANT, and SUGGESTION finding**, create an inline draft note on the specific line.

   All draft comment text MUST be written in **Czech** and in a natural, human-readable tone (not robotic or template-like). Write as a colleague would in a code review.

   **IMPORTANT:** `glab api -F` does not support nested JSON objects — `position[base_sha]` won't be sent correctly and the comment won't appear inline. Use `--input` with a JSON file instead.

   **CRITICAL — Line positioning:** The `new_line` / `old_line` MUST correspond to a line that is **visible in the diff** (a changed line or a context line within a `@@` hunk). If the target line is outside all diff hunks, GitLab returns `line_code: null` and the comment renders as a **duplicate at every hunk boundary** in the file.

   Before creating each comment, **verify the target line is in the diff**:
   - Parse the diff `@@ -old_start,old_count +new_start,new_count @@` headers for the target file
   - Check that your `new_line` falls within `[new_start, new_start + new_count - 1]` of some hunk
   - If the desired line is NOT in any hunk, pick the **nearest visible line** within the same hunk and adjust the comment text to reference the actual target (e.g., "Na řádku 190: ...")
   - For **renamed files** (`old_path` ≠ `new_path`), always double-check hunk ranges — position resolution is less reliable

   Use a Python heredoc to generate the JSON file, then pipe it to glab:

   ```bash
   python3 << 'PYEOF'
   import json

   payload = {
       "note": "**[C1]** <Czech description of the problem and proposed fix>",
       "position": {
           "base_sha": "<base_sha>",
           "head_sha": "<head_sha>",
           "start_sha": "<start_sha>",
           "new_path": "<file>",
           "old_path": "<file>",
           "position_type": "text",
           "new_line": <line_number>
       }
   }

   with open("/tmp/mr<N>_c1.json", "w") as f:
       json.dump(payload, f)
   PYEOF
   glab api projects/:fullpath/merge_requests/<N>/draft_notes \
     --method POST --input /tmp/mr<N>_c1.json -H "Content-Type: application/json"
   ```

   Rules for `new_line` / `old_line`:
   - **Added line** (green in diff): set only `"new_line"`
   - **Deleted line** (red in diff): set only `"old_line"`
   - **Unchanged context line**: set both `"new_line"` and `"old_line"`

3. **Post one summary draft note** (general comment without position) containing **POSITIVE FINDINGS** and **ACTIONS TO TAKE** sections, also written in Czech:

   ```bash
   python3 << 'PYEOF'
   import json

   payload = {
       "note": "## Pozitivni nalezy\n\n- <what is well done>\n- <which patterns are correctly followed>\n\n## Akce k provedeni\n\n- [ ] <action 1>\n- [ ] <action 2>\n\n---\n**Verdict: APPROVE WITH COMMENTS** (X critical, X important, X suggestions)"
   }

   with open("/tmp/mr<N>_summary.json", "w") as f:
       json.dump(payload, f)
   PYEOF
   glab api projects/:fullpath/merge_requests/<N>/draft_notes \
     --method POST --input /tmp/mr<N>_summary.json -H "Content-Type: application/json"
   ```

4. **Do not publish** — comments stay as drafts. At the end, print:
   > "Draft comments have been created. Open the MR in GitLab, review them, edit/delete as needed, and click 'Submit review'."

## Rules

- **Be specific** — every finding must include filename, line number, and a specific fix proposal
- **Compare with existing code** — if a pattern is established in the codebase, reference it
- **Don't nitpick trivial things** — ignore minor stylistic issues that don't conflict with conventions
- **Positive feedback** — always highlight what is well done
- **Skill instructions are in English** for Claude's comprehension, but **all user-facing output and draft comments are in Czech** written in a natural, human tone

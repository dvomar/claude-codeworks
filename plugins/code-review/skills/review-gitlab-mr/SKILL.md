# Review GitLab MR

Performs a comprehensive 3-pass code review of a GitLab merge request directly in the terminal.

## Usage

```
/review-gitlab-mr <MR_number>
```

Example: `/review-gitlab-mr 166`

## Instructions

This skill performs a review in 7 steps.

### Step 1: Load MR metadata and diff

Run these commands via Bash tool:

```bash
glab mr view <N>
```

```bash
glab mr diff <N>
```

From the metadata, note: title, author, source and target branch.

### Step 2: Identify changed files and load their source code

Extract the list of changed files from the diff. For each changed file, read its **complete current content** using Read tool. This is critical — review cannot be done from diff alone, you need to see the context around changes.

### Step 3: Load relevant conventions from the knowledge base

Conventions are provided via CLAUDE.md and MEMORY.md (auto-injected into context).
For detailed conventions, Read these knowledge files generated by `/analyze-codebase`:
- `.claude/knowledge/tech-stack.md` — dependencies, versions
- `.claude/knowledge/architecture.md` — project structure, file placement
- `.claude/knowledge/backend.md` — API, data access, auth, validation patterns
- `.claude/knowledge/frontend.md` — components, styling, state, i18n patterns (if exists)
- `.claude/knowledge/conventions.md` — naming, formatting, testing conventions

If knowledge files don't exist, run `/analyze-codebase` first.

Also read `CLAUDE.md` from the project root for general conventions.

### Step 4: Find similar existing code in the codebase

For each changed file, use Glob and Grep to find **similar existing files** in the same folder or layer (e.g., other controllers, other services, other repositories). Comparison with existing code is key for identifying inconsistencies.

**Limit for large MRs:** If the MR changes **more than 10 files**, search for similar code only for files where CRITICAL or IMPORTANT findings were found in the first quick pass (Pass 1). For MRs with 10 or fewer files, search for all changed files.

### Step 5: Perform 3-pass review

#### Pass 1: Conventions and style

- Naming (variables, methods, classes, files) — consistent with project conventions?
- File organization — correct placement in project structure
- Import/dependency management — consistent ordering, no unused imports
- Formatting — indentation, braces, line endings per project style
- Registration/configuration — follows established patterns

#### Pass 2: Design and architecture

- Dependency management — correct usage, proper scoping/lifetime
- Established patterns — does new code follow patterns found in similar existing files?
- SOLID principles — especially Single Responsibility and Dependency Inversion
- Async patterns — correct usage, proper error propagation
- Comparison with similar existing code — do MR authors do it the same way as elsewhere in the codebase?
- Layering — proper separation of concerns between layers

#### Pass 3: Quality and performance

- DRY — duplicate code, missing extraction of shared logic
- Code smells — overly long methods, too many parameters, god class
- Performance — N+1 queries, unnecessary allocations, missing optimizations
- Security — injection vulnerabilities, improper use of user input
- Readability — is the code understandable without additional context?
- Error handling — consistent with existing patterns

### Step 6: Output structured results

Format output as follows:

```
# Review MR !<N>: <title>
Author: <author> | Branch: <source> -> <target>
Changed files: <count>

---

## CRITICAL (must be fixed before merge)

### [C1] <filename>:<line> — <brief description>
**Problem:** <what is wrong and why>
**Proposed fix:**
```<language>
<specific fix code>
```

---

## IMPORTANT (should be fixed)

### [I1] <filename>:<line> — <brief description>
**Problem:** <what is wrong and why>
**Proposed fix:**
```<language>
<specific fix code>
```

---

## SUGGESTIONS (nice to have)

### [S1] <filename>:<line> — <brief description>
**Suggestion:** <what could be improved and why>

---

## POSITIVE FINDINGS

- <what is well done>
- <which patterns are correctly followed>

---

## ACTIONS TO TAKE

- [ ] <specific action 1>
- [ ] <specific action 2>

---

## SUMMARY

| Category  | Count |
|-----------|-------|
| Critical  | X     |
| Important | X     |
| Suggestions | X   |

**Verdict:** APPROVE / APPROVE WITH COMMENTS / CHANGES REQUESTED
```

### Step 7: Draft comments to GitLab MR

At the end of step 6, ask the user: **"Should I post findings as draft comments to the GitLab MR?"**

If the user agrees:

1. **Get `diff_refs`** from the MR for comment positioning:
   ```bash
   glab api projects/:fullpath/merge_requests/<N> | python3 -c "
   import sys, json
   refs = json.load(sys.stdin)['diff_refs']
   print(refs['base_sha'])
   print(refs['head_sha'])
   print(refs['start_sha'])
   "
   ```

2. **For each CRITICAL and IMPORTANT finding**, create an inline draft note on the specific line.

   **IMPORTANT:** `glab api -F` does not support nested JSON objects — `position[base_sha]` won't be sent correctly and the comment won't appear inline. Use `--input` with JSON:

   ```bash
   echo '{
     "note": "**[C1]** <problem description and proposed fix>",
     "position": {
       "base_sha": "<base_sha>",
       "head_sha": "<head_sha>",
       "start_sha": "<start_sha>",
       "new_path": "<file>",
       "old_path": "<file>",
       "position_type": "text",
       "new_line": <line_number>
     }
   }' | glab api projects/:fullpath/merge_requests/<N>/draft_notes \
     --method POST --input - -H "Content-Type: application/json"
   ```

   Rules for `new_line` / `old_line`:
   - **Added line** (green in diff): set only `"new_line"`
   - **Deleted line** (red in diff): set only `"old_line"`
   - **Unchanged context line**: set both `"new_line"` and `"old_line"`

   For generating JSON payload with escaped characters in `note`, use `python3 -c "import json; print(json.dumps(...))"`.

3. **SUGGESTIONS** — add as a single summary draft note (general comment without position):
   ```bash
   glab api projects/:fullpath/merge_requests/<N>/draft_notes \
     -F "note=## Suggestions

   - [S1] <description>
   - [S2] <description>"
   ```

4. **Do not publish** — comments remain as drafts. At the end, print:
   > "Draft comments have been created. Open the MR in GitLab, review them, edit/delete as needed, and click 'Submit review'."

## Rules

- **Be specific** — every finding must include filename, line number, and a specific fix proposal
- **Compare with existing code** — if a pattern is established in the codebase, reference it
- **Don't nitpick trivial things** — ignore minor stylistic issues that don't conflict with conventions
- **Positive feedback** — always highlight what is well done
- **All outputs in English** (unless the user requests a different language)
